Running a Logos Node

Running a Logos blockchain node turned out to be a two-chapter story. The first chapter involved Nix flakes and Railway. The second involved Docker, a cheap VPS, and a dashboard you can actually use. This post covers both, and explains why we ended up where we did.

Chapter One: Railway + Nix

The first version lives at github.com/KindaInsecureBot/nix-railway. The idea was to use Nix flakes to build the Logos node binary reproducibly, then deploy that to Railway. Nix gives you a hermetic build — same inputs, same output, every time. It's genuinely elegant if you're into that kind of thing.

It worked. The flake builds the node cleanly, Railway picks it up, and you get a running node without much ops overhead. For a quick experiment or a temporary deployment, it's fine.

The problem is cost. A Logos node needs to run 24/7 to be useful. Railway pricing is designed for web services and background workers that don't consume a lot of sustained compute — not for something that's always pegged, always syncing, always talking to peers. The bill climbs fast for something that needs to be permanently on.

So we started over with different constraints: cheap, always-on, and actually maintainable by a normal human who doesn't want to SSH into things constantly.

Chapter Two: VPS + Docker

The second version is at github.com/mart1n-xyz/logos-node-vps. It runs on any Ubuntu 24.04 or Debian VPS, uses Docker Compose to manage the node and dashboard, and sets up in about two minutes with a single command.

On the hardware side: providers like Netcup offer a 4 vCore, 8 GB RAM, 256 GB NVMe VPS for around €8.60/month. That's the right price point for something that runs forever. The Logos node is not particularly resource-hungry — it needs stable connectivity and enough disk for chain data, not beefy CPUs.

Setup

SSH into a fresh Ubuntu 24.04 or Debian server and run:

curl -fsSL https://raw.githubusercontent.com/mart1n-xyz/logos-node-vps/master/setup.sh | sudo bash

That's it. The script installs Docker if it's not there, clones the repo to /opt/logos-node, builds the containers, and starts everything up. The node starts syncing, the dashboard comes up on port 3001.

We originally built this for Ubuntu 24.04. It broke on Debian Trixie due to some package naming differences in the Docker install path. That's since been fixed — the install script now handles both distributions cleanly.

The Dashboard

The part of this we're most pleased with is the dashboard. Once the node is running, open your browser to http://<your-vps-ip>:3001 and you get a web UI with everything you need to actually operate the node:

  • Status tab — node sync state, uptime, current block, connection health
  • Peers tab — live peer list with connection details
  • Keys tab — wallet keys for the node identity
  • Logs tab — live streaming node logs, no SSH required
  • Settings tab — bootstrap peer editor, password protection toggle, and software update

The Settings tab is worth calling out specifically. Bootstrap peers — the initial nodes your node connects to when joining the network — can be edited directly from the browser. No config file editing, no restart scripts. You can also enable password protection on the dashboard from here, which you should do if your VPS IP is public and you don't want anyone who finds port 3001 to have full visibility into your node.

The goal with the dashboard was to make it so that after the initial setup, you rarely need to SSH in again. Checking on the node, watching logs during a network event, updating bootstrap peers — all of that happens in the browser.

Staying Updated

The Settings tab has a Software Update card that shows your current version alongside the latest available. When an update is out, you click Update Now — and that's it. The dashboard triggers a git pull and docker compose up --build on the host, the container restarts, and the page comes back on the new version. The whole thing takes under a minute; the only noticeable effect is a brief moment where the dashboard is unreachable while the container is rebuilding.

Behind the scenes, setup.sh installs a lightweight systemd service called logos-node-updater on the host. It listens on localhost only and handles the actual git and Docker operations. The dashboard proxies update requests to it, which is what keeps the update logic off the container and on the host where it belongs.

If you prefer to do it manually — or if something goes sideways and you want a direct path — the SSH fallback still works: 

cd /opt/logos-node
git pull
docker compose up -d --build

Same result, just without clicking a button.

For bootstrap peers and password changes, as noted above, those go through the Settings tab in the dashboard — no SSH required.

Why This Approach

The Nix + Railway approach was technically interesting but operationally awkward for long-running infrastructure. Nix is great for builds, less great when you need your deployment environment to also be your ops interface.

Docker Compose on a VPS is boring in the right way. Everyone knows how it works, the mental model is simple, and the tooling is stable. Wrapping it in a dashboard that handles the day-to-day removes the remaining friction. Most people running a node don't want to become ops engineers — they want to contribute to the network and not think about it much after setup.

The one-liner install plus browser dashboard is the closest we've gotten to that for the Logos node specifically.
